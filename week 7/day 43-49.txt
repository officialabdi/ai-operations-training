Day 43 — Error Handling in Make.com
What Was Covered
How to make a Make.com scenario handle failures without silently breaking or stopping completely.

Key Concepts
What error handling is
When a module in Make.com fails, by default the entire scenario stops and marks the run as an error. Error handling means adding instructions that tell Make what to do instead — so the scenario can recover, notify you, or skip the failed bundle and keep going.
Why it matters for client work
A scenario that silently fails is worse than one that never existed — the client assumes it's running when it isn't. Professional automations catch failures, log them, and notify someone. That's what makes a delivery production-ready.
The five error handlers in Make.com

Break — stops the run, saves it as an incomplete execution so you can fix and retry it later. Requires "Allow storing of incomplete executions" to be turned on in scenario settings.
Ignore — skips the failed bundle and continues processing the rest. Use for non-critical steps where losing one bundle is acceptable.
Resume — continues the scenario using a substitute value you define in the handler settings. The scenario treats the failed module as if it returned that value.
Commit — marks the run as successful and stops at the point of failure. Only useful for ACID/database modules.
Rollback — marks the run as an error and attempts to undo database changes made earlier in the run. Only works on ACID modules. Cannot unsend emails or undo file deletions.

How to attach an error handler
Right-click the module you want to protect → Add error handler → choose the handler type. The error handling path appears as a dotted line leading away from that module. Modules on this path only run when an error occurs.
The error handling path
The error handler must always be the last module on the error path — nothing can come after it. You can add other modules before it on the path (e.g. a Google Sheets logger) and the handler closes the path.
What the error path has access to
Inside the error path, you can access the error message using Error: Message from the mapping panel. This lets you log what went wrong.

Build
Added an error handler to the Luigi pipeline. When a module fails, the error is caught rather than crashing the run.

Key Takeaway
Anyone can build an automation that works when everything goes right. Production-ready automations handle when things go wrong — the client never loses data silently and always knows if something breaks.


Day 44 — Scheduling & Cron Triggers
What Was Covered
How to make a Make.com scenario run automatically on a time-based schedule instead of waiting to be triggered by an event.

Key Concepts
What a schedule trigger is
A trigger that wakes the scenario up at a set time or interval and runs it automatically. No webhook, no manual trigger, no human action required.
Webhook trigger vs schedule trigger

Webhook: fires the instant something happens (form submission, payment, API call)
Schedule: fires when a clock hits a time
Use webhooks for event-driven tasks. Use schedules for time-driven recurring tasks.

When to use a schedule
When the task needs to happen on a rhythm regardless of external events. Examples: daily sales summary, hourly stock check, weekly report, monthly invoice run.
How to set a schedule in Make.com

Click the clock icon on the trigger module
Choose Schedule as the trigger type
Set a simple interval (every X minutes/hours/days) or an advanced cron expression
Turn the scenario ON

What a cron expression is
A short line of text that describes exactly when something should run. Has 5 positions separated by spaces:
[minute] [hour] [day of month] [month] [day of week]
Cron position values

Specific number — run at exactly this value
* — every (no restriction on this position)
1-5 — range (from value 1 to value 5)
1,3 — list (on these specific values only)

Day of week numbering
1 = Monday, 2 = Tuesday, 3 = Wednesday, 4 = Thursday, 5 = Friday, 6 = Saturday, 0 or 7 = Sunday
Common cron expressions

0 18 * * 1-5 — every weekday at 6pm
0 * * * * — every hour on the hour
0 8 * * 1 — every Monday at 8am
0 8 1 * * — first day of every month at 8am

Three questions to choose the right schedule for a client

How fresh does the data need to be?
When is the output actually useful to the client?
How often does the underlying data change?


Build
Updated the Luigi pipeline trigger from webhook to a scheduled cron trigger. Set to run every weekday at 6pm: 0 18 * * 1-5.

Key Takeaway
Set it once and it runs forever. Most recurring business tasks belong on a schedule, not a webhook.


Day 45 — Data Transformation in No-Code
What Was Covered
How to reshape data inside Make.com so it arrives at the destination in the correct format — no manual cleanup required.

Key Concepts
What data transformation is
Taking data in the shape it arrives and changing it into the format you actually need before passing it on. The form sends 2025-03-14T09:30:00.000Z. The sheet needs 14/03/2025. That gap is where transformation lives.
Why it matters for clients
Every system speaks a slightly different dialect. Without transformation in the middle, staff do manual cleanup every day. The automation cleans it before it touches the destination — nobody ever has to fix it manually.
The four transformation types

Mapping — routing a value from one module to another with no change to the value itself
Formatting — changing how a value looks without changing what it means (date format, text case, number rounding)
Filtering — deciding whether a bundle should continue based on a condition
Aggregating — combining multiple bundles into one (e.g. summing totals across all orders)

Where transformation happens in Make.com
Inside the field mapping of the receiving module — not in a separate transform module. When you configure a Google Sheets module, you wrap the incoming variable in a function inside the field. The transformation happens at the point of handoff.
Make.com built-in functions — text

upper(text) — converts to ALL CAPS
lower(text) — converts to all lowercase
capitalize(text) — capitalises the first letter of the first word only
trim(text) — removes leading and trailing whitespace
replace(text; search; replacement) — finds and replaces a string

Make.com built-in functions — numbers

round(number; precision) — rounds to a set number of decimal places
abs(number) — removes negative sign
formatNumber(number; decimalMark; thousandsSeparator) — controls display format

Make.com built-in functions — dates

formatDate(date; format) — takes a raw date and outputs it in the format you specify
Format tokens: DD = day, MM = month, YYYY = four-digit year, HH:mm = hours and minutes

Important: Make.com uses semicolons, not commas
formatDate(1.created_at; "DD/MM/YYYY") ✓
formatDate(1.created_at, "DD/MM/YYYY") ✗
Write it wrong and the function silently fails.
Nesting functions
Functions can be nested. They run from the inside out.
trim(capitalize(lower(1.customer_name))) runs: lower first → capitalize → trim.
Order matters — getting it wrong produces the wrong result.
Make.com capitalize limitation
capitalize() only uppercases the first letter of the first word. It does not produce title case (Luigi Rossi). There is no native title case function in Make.com. This is a known platform limitation.
When to transform in the automation vs the destination
Transform in the automation when:

The destination can't do it reliably
Multiple destinations need the same clean data
The transformation affects whether data should be processed at all
The client might accidentally break the destination's formatting


Build
Extended the Luigi pipeline's Google Sheets module to apply transformations inside the field mapping:

Customer name: trim(capitalize(lower(1.customer_name)))
Order total: round(1.order_total; 2)
Order date: formatDate(1.created_at; "DD/MM/YYYY")

Test submission confirmed transformed data landing correctly in the sheet.


Day 46 — Connecting Multiple Apps
What Was Covered
How to build a Make.com scenario that sends data to more than one destination in a single run.

Key Concepts
What connecting multiple apps means
Adding more destination modules after a single trigger so one event causes actions in multiple tools simultaneously. One form submission → Google Sheets + Gmail + Slack, all in one run.
How Make.com acts as a central hub
Make.com sits between all apps. It receives data from the trigger, and routes it to every connected destination. Each module hands its output to the next module in the chain. You don't need each app to talk directly to every other app.
How data flows through a multi-app scenario
Every module has an output. That output is available to every module that comes after it. Module 3 can use data from Module 1 or Module 2. Modules do not re-fetch data — they pull from what came before them in the chain.
How to add a destination module
Click the + after any existing module on the canvas → search for the app → configure it → map variables from earlier modules into its fields.
Slack connection type — bot vs personal
When connecting Slack in Make.com, choose Slack (bot) not a personal account. Bots post on behalf of the app, not tied to an individual's login. In a client setup you never want automated messages tied to someone's personal account.
Variables must be selected, not typed
In Make.com field mapping, variables must be inserted by clicking them from the mapping panel — they appear as coloured tokens/pills. If you type {{variable_name}} manually, it prints as literal text and does not pull live data.
When to use multiple destinations vs one
Use multiple destinations when:

Different people need to be notified in different ways (customer gets email, staff gets Slack)
The same data needs to live in multiple systems
An action needs to be logged somewhere separate

Keep one destination when:

Only one system needs to know about the event
Adding more creates noise without value
The client hasn't asked for it

The client rule
Every module you add should answer: who needs to know about this, and where do they actually look? Luigi checks Slack. The customer checks their inbox. The spreadsheet is the record. Three destinations, three purposes — justified.

Build
Extended the Luigi pipeline from Tally → Google Sheets → Gmail to include a fourth module: Slack notification to a #luigi-orders channel. Single test submission fired all four apps in one run. Dynamic data (customer name, order details) passed through correctly as mapped tokens.


Day 47 — Monitoring & Logging Automations
What Was Covered
How to track what a Make.com scenario does, when it runs, and whether it succeeded — and how to add a permanent log to any automation.

Key Concepts
What monitoring and logging means in no-code
Monitoring = watching whether your scenario is running and catching problems. Logging = keeping a permanent written record of what happened, when, and what the result was.
Why logging matters for client-facing pipelines
Without logging, if a scenario silently fails or processes wrong data, nobody knows. Logging gives a client a full audit trail — proof the automation ran, what it handled, and when. It's also your first tool for debugging when something goes wrong.
Make.com execution history
Every scenario run is saved in Make.com's execution history. For each run you can see: when it ran, how long it took, how many operations it used, which modules processed which bundles, and any errors that occurred. You access it via the history/clock icon in the scenario builder.
What the execution log shows

Run ID and timestamp
Duration and operations used
Each module and whether it completed or failed
The bundle data that passed through each module
Error messages if a module failed

How to read a failed run
Click the run in history → find the module with the warning icon → click the bubble to see the bundle that caused the error and the error message.
Setting up a logging module in Make.com
Add a Google Sheets → Add a Row module at the end of your scenario (or at the end of each path in a router). Map the following fields to a dedicated Log tab:

Timestamp: {{now}}
Order/record identifier from the upstream data
Status: hardcoded text like Processed successfully

Log tab structure
Create a dedicated tab in the Google Sheet (not the main data tab). Headers: Timestamp, Order ID, Status. Every successful run appends a new row.
The difference between execution history and a log module

Execution history: lives inside Make.com, visible only to you, retained for a limited time depending on your plan
Log module: writes to a Google Sheet the client can see, permanent, part of your deliverable

Client value of logging
"Every time the automation runs, it logs what happened, when it happened, and whether it succeeded. If something goes wrong, you know immediately and you have a full record to investigate."

Build
Added a Google Sheets logging module to the end of the Luigi pipeline. Every run writes a timestamped row to a dedicated Log tab. Confirmed log entry appearing correctly after a test submission.

Key Concepts
What a real client engagement looks like

How to read a client brief and extract a build spec
From a plain-English brief you identify:

What the outputs are (emails, logs, notifications)
What happens when something goes wrong

The Brennan's Bakery scenario built

Data source: Google Sheets — reads today's orders

Path A (standard orders) → Gmail confirmation to customer
Path B (fallback for flagged/problem orders) → separate Gmail with different message


Logger: Google Sheets Add a Row → writes a timestamped record after each successful run
Error handling: catch failures and write to an Errors tab

Router fallback path
In Make.com a Router can have a fallback path — a path that catches any bundle that didn't match any other filter condition. This is set by turning on "Set route as fallback: Yes" in the path's filter settings. Use it to catch unexpected data or edge cases rather than letting them drop silently.
Text Aggregator module
A module that combines multiple bundles into a single piece of text. Used when you want to summarise multiple rows from a sheet into one email rather than sending one email per row.
Designing before building
Before opening Make.com: map the full flow in plain English. Trigger → data → logic → output → error handling. Every module has a defined purpose before you touch the canvas. This is how professional builds start.
Client handover
At the end of a build you produce a plain-English explanation of what the automation does, what each part is for, and how to interpret the logs. This is what you hand to the client, not a technical spec.

Modules Built
Module 1 — Schedule Trigger
Wakes the scenario on a set schedule.
Set to daily at 08:00. Left on manual trigger during testing.
Module 2 — Google Sheets: Search Rows
Fetches rows from the Orders tab.
Filter on date column: {{formatDate(now; "DD/MM/YYYY")}}
Returns only today's orders. Each matching row = one bundle.
Max results: 100.
Module 3 — Router (3 paths)
Splits bundles based on order type.

Path 1 — Delivery: order type (D) → Text operators: Equal to → delivery
Path 2 — Collection: order type (D) → Text operators: Equal to → collection
Path 3 — Invalid Email: catches bundles with bad email format before Gmail

Module 4 — Gmail: Send an Email (Delivery path)
Sends a personalised delivery confirmation to the customer.
All fields mapped from the Google Sheets bundle.

To: {{email (C)}}
Subject: Your Luigi's Pizzeria delivery is confirmed!
Body: Customer name, order items, total, 45-minute delivery estimate

Module 5 — Gmail: Send an Email (Collection path)
Same structure as delivery. Different message.

To: {{email (C)}}
Subject: Your Luigi's Pizzeria order is ready to collect!
Body: Customer name, order items, total, collect within 30 minutes

Module 6 — Error Handler (Router path 3)
Catches bundles with invalid email addresses at the Router level.
Logs the failed order to the Errors tab before Gmail is ever reached.
Fields logged:

Timestamp: {{now}}
Order ID: {{1. order id (A)}}
Customer Name: {{1. customer name (B)}}
Error Message: invalid email address — skipped

Note: Regex filter for email format validation to be confirmed separately.
Module 7 — Logger (both paths)
After each successful Gmail send, writes a row to the Log tab.
Both delivery and collection paths have their own logger.
Fields logged:

Timestamp: {{now}}
Order ID: {{1. order id (A)}}
Status: Processed successfully


Key Concepts Applied

Dynamic date filtering in Google Sheets using formatDate(now)
Multi-path routing with conditional filters
Variable mapping from upstream modules into email body content
Error handling at the Router level — catch bad data before it hits Gmail
Logging successful operations for audit trail
Building and testing module by module before connecting


What I Learned About Error Handling in Make

Resume handler: replaces a failed module's output with a substitute value — wrong tool for logging errors
Ignore handler: skips a failed bundle and continues the run — correct for non-critical failures
Error handlers only catch failures on the module they are attached to
Gmail does not validate email addresses — it will attempt to send to anything
The cleanest approach for data validation is catching bad data at the Router before it reaches Gmail, not after Gmail fails
Make's email format validation only checks structure (@ symbol, domain) — it cannot verify if an address actually exists
Full email delivery verification requires an external API service (ZeroBounce, NeverBounce) — this is a billable add-on
